# deep-researching

## 定位与描述

**deep-researching** 是一个“研究助手型”工作流：像研究助手一样做信息调研——**检索 → 阅读 → 综合 → 对照核验**，必要时再升级到更系统的调研方式。用于需要“查资料 / 核验事实 / 做研究梳理或报告”的问题。

本 skill 面向本仓库提供的 Perplexity MCP 工具：

- `perplexity_search`
- `perplexity_ask`
- `perplexity_reason`
- `perplexity_research`（重型）

## 何时使用

### 适用场景

- 需要**带来源链接/引用**的回答（例如写报告、做资料汇总、要能回溯证据）。
- 需要在多个来源之间**交叉验证**（例如不同机构口径不一致、存在争议结论）。
- 需要**系统性调研**与结构化输出（例如综述、对比分析、调研提纲、风险清单）。

### 不适用场景（建议不用或先不用）

- 纯主观创作/脑暴（没有“需要引用”的要求）。
- 只需一个简单事实且不追求来源（可直接问答）。
- 低风险、低成本的快速查询（优先用一次 `perplexity_search` 即可）。

## 总体调用策略（强约束）

1. **遇到不确定/不了解的信息，优先调用 `perplexity_search` 或 `perplexity_ask` 获取资料**，不要凭空编造。
2. **需要系统性调研时才使用 `perplexity_research`**（重型、慢、成本高），并优先在 search/ask 无法覆盖时再升级。
3. **一个完整问题应在同一个上下文里完成**：对同一研究主题的 follow-up 调用，必须复用同一个 `backend_uuid`。
4. **尽量不要高频调用**：尤其避免频繁 `perplexity_research`；能在一次 query 查清楚的，就把子问题合并到一次 query。

## 上下文与续问（backend_uuid）规则

### 如何复用同一上下文

- 第一次调用不带 `backend_uuid`（新对话）。
- 从工具返回中读取 `structuredContent.backend_uuid`（如有），并在后续相关调用中把它作为 `backend_uuid` 传回去，形成同一对话线程的续问。

### 何时必须重置（不要复用旧 UUID）

- 研究主题发生明显切换（例如从“政策解读”切到“产品对比”）。
- 你需要避免上下文污染（例如前一轮引入了错误假设/错误定义）。

## 工具选择指南（如何使用该 MCP）

### 1) `perplexity_search`：默认起手式（广度 + 线索）

用于：快速覆盖多个子问题、拿到候选来源、建立调研地图。

推荐做法：

- 把多个子问题合并到一个 `query` 里（列点即可）。
- 一般不需要在每次 `query` 里反复强调“要引用/要链接”（Perplexity 通常会附带来源）；当需要精确溯源或对照核验时，再通过 follow-up 补齐你要的格式与细节。

### 2) `perplexity_ask`：补齐细节（窄而深）

用于：在已有线索后，对某个点做更聚焦的追问。

常见用法：

- 针对 search 给出的某条结论，追问“依据是什么？边界条件/适用范围是什么？”
- 当需要精确对照核验时，再追问“把关键来源逐条列出来，并说明各自支持的结论点”。

### 3) `perplexity_reason`：组织推理与结构化产出

用于：把已有材料组织成更清晰的框架（例如对比表、论证链、风险与建议）。

注意：

- 不要把推理当作新事实来源；如出现关键不确定点，优先回到 search/ask 补齐信息再总结。
- 如需减少模型推理痕迹，可使用 `strip_thinking`（可选）。

### 4) `perplexity_research`：系统性调研（重型、谨慎使用）

用于：search/ask 无法覆盖的系统性调研（例如需要更全面的搜集、跨多来源综合、形成报告）。

纪律：

- 调用前先把目标与输出结构写清楚（避免多轮反复）。
- 能用 1 次 research 解决的，就不要拆成多次。
- research 之后优先用同一 `backend_uuid` 做 1 次 follow-up：核验争议点或补齐你需要的结构化细节，而不是再开新 research。

## 推荐调研流程（可直接照做的步骤）

1. 明确交付物：要回答什么、输出形式（要点/表格/报告）、时间范围、地区范围、可信度要求。
2. 用一次 `perplexity_search` 做“调研地图”：
   - 合并子问题；
   - 要求指出争议点与需进一步核验的点。
3. 阅读并筛选：优先原始/权威来源（标准/论文/政府/官方文档），区分一手与二手解读。
4. 针对缺口用同一 `backend_uuid` 做 follow-up：
   - 若有冲突：追问“分别给出来源 A/B 的原文依据与差异点，并判断更可信的原因”。
5. 仅当仍不足时升级到 `perplexity_research`（同样要求带引用，并尽量一次问清）。
6. 最终输出：
   - 结论（分点）；
   - 冲突与不确定性说明（哪些点未被充分证实、哪些点来源口径不一致）。

## 输出建议（默认）

- 应区分：
  - “事实/来源支持的陈述”
  - “推断/分析”
  - “不确定/待核验”
- 当不同来源结论不一致时，应：
  - 列出差异点；
  - 说明你采用哪个结论及理由（例如来源权威性、时间新旧、是否为一手材料）。

## 安全与合规提示

- 不要在任何输出中泄露 Cookies、token、账号信息或其他敏感数据。
- 引用链接时避免包含带鉴权的私有链接或任何可能暴露用户身份的内容。
